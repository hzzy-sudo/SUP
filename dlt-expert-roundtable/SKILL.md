# 大乐透专家圆桌会议 Skill

## 核心理念

传统的专家预测是各自独立分析，导致基于相同数据产生高度一致的结果。圆桌会议模式引入**对抗性协作机制**，让专家们：
1. 先各自提出初步观点
2. 互相质疑和挑战
3. 在辩论中调整立场
4. 最终达成经过充分讨论的共识

这种模式模拟真实专家团队的决策过程，通过思想碰撞产生更有价值的推荐。

## 使用场景

- 用户希望获得经过多角度验证的推荐结果
- 用户想了解不同分析方法之间的冲突与协调
- 用户需要看到专家推荐背后的推理过程和争议点
- 用户希望通过专家辩论来评估推荐的可靠性

## 会议流程设计

### 第一阶段：独立分析（5分钟）

每位专家基于自己的方法论独立分析，提出：
- 前区5个核心号码
- 后区2个核心号码
- 关键论据（不超过3条）

**输出格式**：
```
【专家名】初步观点：
前区：XX XX XX XX XX
后区：XX XX
核心论据：
1. [论据1]
2. [论据2]
3. [论据3]
```

### 第二阶段：交叉质疑（10分钟）

专家们互相质疑对方的推荐，重点关注：
- 数据解读的合理性
- 逻辑推理的严密性
- 与历史规律的一致性
- 不同方法之间的冲突

**质疑规则**：
1. 每位专家必须质疑至少1位其他专家
2. 质疑必须基于数据和逻辑，不能主观臆断
3. 被质疑者必须回应并调整或坚持观点

**输出格式**：
```
【专家A】质疑【专家B】：
[具体质疑内容]

【专家B】回应：
[回应内容]
[是否调整观点]
```

### 第三阶段：阵营协商（8分钟）

三个阵营内部协商，形成阵营共识：
- 保守派：强调热码延续性
- 激进派：强调冷码回补
- 平衡派：寻找折中方案

**输出格式**：
```
【保守派共识】
前区核心：XX XX XX XX XX
后区核心：XX XX
阵营理由：[理由]

【激进派共识】
前区核心：XX XX XX XX XX
后区核心：XX XX
阵营理由：[理由]

【平衡派共识】
前区核心：XX XX XX XX XX
后区核心：XX XX
阵营理由：[理由]
```

### 第四阶段：全体投票（5分钟）

对所有被提名的号码进行投票：
- 每位专家对前区号码投5票
- 每位专家对后区号码投2票
- 统计得票数，形成热度排名

**输出格式**：
```
【前区号码投票结果】
08号：7票（陈冰、江川、万妙仙、陈涛普、富哥、陈樱芳、文新林）
22号：6票（陈冰、江川、陈涛普、富哥、陈樱芳、海天）
...

【后区号码投票结果】
08号：5票（陈冰、江川、万妙仙、陈涛普、富哥）
04号：5票（陈青峰、海天、文新林、陈樱芳、江川）
...
```

### 第五阶段：共识达成（7分钟）

基于投票结果和辩论过程，形成最终推荐：
- 15+5大复式：高票号码组合
- 9+3小复式：核心高票号码
- 5+2单注：专家一致认可的号码

**共识原则**：
1. 前区号码至少获得3票才能进入大复式
2. 前区号码至少获得5票才能进入小复式
3. 前区号码至少获得7票才能进入单注
4. 后区号码至少获得2票才能进入大复式
5. 后区号码至少获得4票才能进入单注

## 核心分析逻辑

### 对抗机制设计

**1. 数据视角对抗**
- 短期派 vs 长期派：最近10期 vs 最近100期
- 热码派 vs 冷码派：追热 vs 补冷
- 形态派 vs 遗漏派：形态跟随 vs 遗漏回补

**2. 权重分配对抗**
- 保守派：热码权重0.7，冷码权重0.3
- 激进派：热码权重0.3，冷码权重0.7
- 平衡派：热码权重0.5，冷码权重0.5

**3. 逻辑推理对抗**
- 归纳派：从历史数据归纳规律
- 演绎派：从理论模型演绎结果
- 直觉派：基于经验和盘感判断

### 质疑触发条件

当出现以下情况时，必须触发质疑：
1. 某个号码被超过6位专家同时选中
2. 某个号码的遗漏期数超过30期但无人选择
3. 某位专家的推荐与其历史风格明显不符
4. 不同方法得出完全相反的结论
5. 某个论据存在明显的数据错误

### 共识形成机制

**投票权重**：
- 基础权重：每位专家1票
- 历史准确率加权：近10期命中率 × 0.5
- 论据质量加权：被其他专家认可的论据数 × 0.3
**争议解决**：
当某个号码得票数在阈值边缘时（如4-5票），采用以下规则：
1. 查看该号码在多个维度的支撑度
2. 评估支持该号码的专家的历史准确率
3. 考虑该号码与其他高票号码的协调性
4. 最终由会议主持人（系统）裁决

## 输出规范

### 必须包含的章节

1. **会议信息** - 时间、期号、参会专家
2. **第一阶段：独立分析** - 9位专家的初步观点

### 质量标准

**数据引用**：
- 所有论据必须引用具体数据
- 数据来源必须明确（如"460期条件转移"、"历史同期"）

- 必须体现不同方法论的差异

## 风险与限制

### 已知限制

1. **时间成本**：完整的圆桌会议流程需要较长的生成时间
2. **复杂度**：需要模拟9位专家的思维过程和互动
3. **一致性**：虽然引入对抗机制，但基于相同数据仍可能趋同

### 风险提示

1. **不保证准确性**：圆桌会议只是决策过程的模拟，不改变彩票的随机性
2. **可能过度复杂**：对于简单需求，传统单专家预测可能更高效
3. **共识偏差**：多数人的共识不一定正确，可能存在群体性盲区

## 使用建议

### 适用场景

✅ 重要期次，需要慎重决策
✅ 希望了解不同方法的优劣
✅ 需要评估推荐的可靠性
✅ 学习专家分析的思维过程

### 不适用场景

❌ 时间紧迫，需要快速结果
❌ 只需要简单的号码推荐
❌ 不关心推理过程
❌ 计算资源有限

### 最佳实践

1. **定期召开**：建议每5期召开一次圆桌会议
2. **记录历史**：保存会议记录，用于评估专家可信度
3. **结合使用**：圆桌会议结果可与单专家预测对比
4. **持续优化**：根据历史准确率调整专家权重

## 技术实现说明

### 执行方式

本skill可通过以下方式执行：

**方式1：手动执行**
```bash
# 在Kiro中直接调用
请按照 dlt-expert-roundtable skill 为大乐透第26033期召开专家圆桌会议
```

**方式2：脚本集成**
```python
# 在 run_prediction_suite.py 中添加圆桌会议模式
python supser/scripts/run_prediction_suite.py --mode roundtable --issue 26033
```

### 数据依赖

- `dlt-support.json` - 共享统计数据
- `snapshot-support.json` - 当期快照
- 各专家的 `SKILL.md` - 方法论参考
- 各专家的历史推荐记录（可选）

### 输出格式

- Markdown格式的会议记录
- JSON格式的投票统计
- 标准的推荐结果（15+5、9+3、5+2）
